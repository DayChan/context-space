## 背景

当前 Todo 列表中的勾选标识只是装饰元素；People API 仅返回来源 ID；同步状态只在开始和结束时更新；飞书消息查询也没有形成“机器人会话绝不进入分析链路”的完整边界。变更横跨适配器、同步服务、领域写入、HTTP API 和 Web UI，且必须保持飞书只读、工作区 Markdown 为事实来源及现有乐观并发语义。

## 目标 / 非目标

**目标：**

- 让用户能在列表和详情中可靠切换 Todo 完成状态。
- 在持久化之前排除机器人消息和机器人 P2P 会话。
- 让 People 详情直接展示来源消息内容。
- 在同步运行期间展示可轮询的细粒度进度和错误。

**非目标：**

- 不向飞书回写任务状态。
- 不使用姓名关键字猜测历史消息是否来自机器人。
- 不引入 WebSocket、任务队列或新的外部依赖。
- 不自动删除无法可靠识别的既有历史数据。

## 技术决策

### Todo 状态使用专用 API

新增受约束的 Todo 状态更新端点，由服务端读取最新文档并原子写入，只允许合法生命周期状态。相比让勾选框调用通用文档编辑接口，该方案缩小写入面、避免前端拼装整个文档和无意覆盖并发字段。原生任务派生文档再次同步时保留既有本地状态，使用户操作不会被只读上游结果重置。

### 机器人过滤采用双层防线

飞书 CLI 查询增加机器人发送者排除参数以减少数据传输；规范化阶段再根据明确的发送者或 P2P 对端类型字段过滤。P2P 对端是机器人时丢弃会话双方的消息，确保用户发给机器人的内容也不进入存储、People 发现或 LLM。只使用结构化类型字段，不根据显示名推断。

### Provenance 在 API 边界解析

People 详情 API 按人物文档中的引用从现有索引解析 `source` 文档，返回有限的标题、正文、时间和来源类型。UI 不直接推导文件路径或二次批量请求，这保证引用解析规则集中、避免 N+1 请求，并只暴露该人物实际引用的来源。

### 同步进度使用内存实时态与持久化终态

`LarkSyncService` 在每个来源、时间窗口、分页和分析阶段更新结构化进度；专用只读状态 API 返回当前快照。UI 在运行中短轮询并展示状态窗口。高频页级进度不反复写 Markdown，开始、重要阶段和最终结果仍持久化，降低磁盘写放大并保持重启后的最后运行状态。

## 风险 / 权衡

- [旧 CLI 不支持机器人过滤参数] → 本地规范化仍是安全边界，并让命令错误在来源状态中明确可见。
- [运行进程重启会丢失页级实时进度] → 持久化的开始状态和最终状态仍可恢复；启动加载时将遗留运行态标记为中断。
- [旧数据缺少发送者类型，无法确定是否为机器人] → 不做不可靠的名称推断；后续同步从入口阻断，历史清理留给有明确身份数据的迁移。
- [状态轮询增加少量请求] → 仅在运行中轮询专用轻量端点，完成后停止。

## 迁移计划

数据结构读取兼容缺少 `progress` 的旧状态文件；首次新同步会写入新字段。部署后无需批量迁移。回滚时新增字段会被旧版本忽略，Todo 和来源 Markdown 格式保持兼容。
