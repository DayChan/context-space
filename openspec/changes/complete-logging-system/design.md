## 背景

Context Space 当前通过 Markdown 保存同步状态与分析运行结果，但服务端只在启动时执行两次 `console.log`。HTTP 请求、飞书分窗口拉取、`lark-cli` 子进程、LLM 批次、Provider 诊断、校验和写入之间没有统一事件，也没有能够贯穿这些阶段的关联 ID。系统处理的消息和人物数据具有隐私属性，因此日志必须默认可用，同时严格避免把业务正文和认证信息变成第二份数据副本。

本次变更横跨服务入口、Express、飞书适配器、分析协调器和 Provider 调用，运行环境是 Node.js 20 及以上版本。日志保存在工作区私有目录，`workspace/` 默认不进入 Git。

## 目标与非目标

**目标：**

- 提供类型明确、可注入、可测试的统一 Logger，并让每条日志成为独立 JSON 对象。
- 在正常开发和生产运行中同时输出到终端与本地文件。
- 支持等级过滤、日期与大小轮转、保留期清理、显式 `flush` 和 `close`。
- 自动合并组件字段与异步调用上下文，使请求、同步和分析事件可串联。
- 记录足以定位故障的错误类型、脱敏消息、堆栈、耗时、计数、事件类型和 Provider 诊断。
- 通过字段拒绝、字符串脱敏、深度与长度限制，确保原始消息、Prompt、完整模型响应和凭证不会进入日志。
- 日志文件不可写时让核心业务继续运行，并通过安全的应急事件提示输出故障。

**非目标：**

- 不引入远程日志采集、云端 APM、指标数据库、分布式追踪后端或告警平台。
- 不在 Web 界面中实现日志浏览器。
- 不记录请求体、查询参数值、飞书响应正文、LLM Prompt、模型完整响应或环境变量快照。
- 不改变现有同步、分析和 Markdown 状态记录的业务语义。

## 技术决策

### 使用 Node.js 标准库实现轻量 Logger

新增 `src/logging/`，定义 `Logger` 接口、日志等级、字段类型、配置解析、递归脱敏、异步上下文和 JSONL 文件目标。文件输出使用串行 Promise 队列与 `appendFile`，不在请求路径中等待磁盘完成；`flush` 与 `close` 等待队列排空。

当前日志量低且仅本地单进程运行，标准库实现可以避免新增依赖、额外传输和复杂配置。备选方案是 Pino 加第三方轮转组件，但其主要吞吐优势在本项目当前规模下不足以抵消依赖和双重配置成本。

### 采用稳定事件信封

每条日志至少包含 `timestamp`、`level`、`service`、`event` 和 `pid`。组件 Logger 通过 `child` 增加 `component`，`AsyncLocalStorage` 自动合并当前的 `request_id`、`sync_id`、`run_id`、`batch_id` 或 `source_id`。调用点只提交结构化计数、状态、耗时和安全标识符，不拼接面向人类的自由文本日志。

字段优先级为调用字段、异步上下文、子 Logger 字段和基础信封；保留字段不能被调用方覆盖。

### 终端与文件采用同一份 JSON

终端和文件接收同一条序列化后的 JSON。默认等级为 `info`，默认启用终端与文件；Vitest 环境默认静默以避免污染测试输出，但显式配置仍可开启。

日志目录默认为 `<workspace>/.context/logs/`。文件使用 UTC 日期命名为 `context-space-YYYY-MM-DD.jsonl`；达到大小上限后使用递增分段 `context-space-YYYY-MM-DD.N.jsonl`。启动和日期切换时清理超过保留天数的匹配文件，不删除目录中的其他文件。目录和新文件分别使用仅当前用户可访问的权限。

### 通过环境变量配置而不写入 Markdown

日志配置在进程启动前生效，使用 `CONTEXT_SPACE_LOG_LEVEL`、`CONTEXT_SPACE_LOG_CONSOLE`、`CONTEXT_SPACE_LOG_FILE`、`CONTEXT_SPACE_LOG_DIR`、`CONTEXT_SPACE_LOG_MAX_BYTES` 和 `CONTEXT_SPACE_LOG_RETENTION_DAYS`。无效值回退到安全默认值并产生配置警告事件，不因日志配置阻止服务启动。

日志目录可以显式放到工作区外，便于进程管理器采集；默认仍位于私有工作区。日志配置不进入 `config/*.md`，避免运行环境设置被规范数据索引。

### 脱敏发生在序列化边界

Logger 在任何目标接收数据前统一净化字段。认证头、Cookie、密码、Secret、常见访问令牌、Prompt、正文、标准输入输出和完整响应等字段名直接替换为 `[已脱敏]`；字符串中的 Bearer、API Key、Codex 会话令牌等模式也会替换。Error 被转换为名称、脱敏消息、脱敏堆栈、错误码和有限深度的 cause。

对象使用循环引用保护，数组、对象深度和字符串长度均有限制。业务调用点仍承担数据最小化责任，Logger 脱敏作为第二道防线。

### 在边界和阶段转换处记录事件

- Express 中间件生成或接受格式安全的 `x-request-id`，设置响应头，并在完成或中断时记录方法、路径模板或路径、状态与耗时；不记录 URL 查询值和请求体。
- 全局错误处理中记录分类、状态和异常；入口记录启动、监听、停止、未捕获异常与未处理拒绝。
- 飞书同步记录本轮配置、各来源与窗口的开始/完成、数量、问题分类、批量分析摘要和最终状态。`lark-cli` 只记录命令族、参数名、耗时、输出字节数和结构化 issue，不记录完整 stdout/stderr。
- 分析协调器记录批次选择、跳过、Provider 可用性、调用完成、usage、事件类型、校验、写入与失败。`ProviderAnalysisResponse.diagnostic` 经现有安全清理后以独立 warning 事件持久化。

### 通过依赖注入保持测试与扩展能力

`createApp`、`LarkSyncService`、`LarkCliCommandRunner` 和 `AnalysisCoordinator` 接受可选 Logger；未传入时使用统一默认实例或无副作用 Logger。测试使用内存目标捕获 JSON 行，不依赖真实文件、飞书或 LLM。

## 风险与取舍

- [日志队列在大量突发事件下占用内存] → 当前只记录阶段事件而非每条正文或每个流式 Token，并设置字段大小限制；关闭时等待队列排空。
- [文件写入失败导致诊断缺失] → 文件目标失败后在 stderr 输出一次脱敏应急事件，继续保留终端日志并允许业务运行。
- [脱敏规则误删有用字段] → 采用明确字段集合而非模糊匹配，例如保留 `input_tokens` 等 usage 计数；测试凭证、正文、Prompt 与正常标识符。
- [脱敏规则无法覆盖未知秘密格式] → 调用点默认不提交原始载荷，字符串模式只作为第二层防护，并在 README 明确隐私边界。
- [异步上下文增加隐式行为] → 仅合并小型关联字段，Logger 仍支持显式字段；测试并发请求互不串号。
- [按日期和大小轮转不是跨进程安全的] → 当前架构只支持单服务进程写入同一目录；文档明确不要让多个进程共享同一日志文件前缀。

## 迁移方案

1. 新增 Logger 与独立测试，不改变业务模块。
2. 在 `createApp` 和入口接入 Logger，启用请求、生命周期和全局错误事件。
3. 依次为飞书同步、CLI Runner、分析协调器和 Provider 诊断增加事件。
4. 补充 README 配置与查询示例，运行完整检查。
5. 发布后首次启动自动创建 `.context/logs/`；不迁移或删除旧的 Markdown 状态记录。

回滚时可设置 `CONTEXT_SPACE_LOG_FILE=false` 与 `CONTEXT_SPACE_LOG_CONSOLE=false` 立即关闭输出；代码回滚只会遗留可安全手动删除的 `.jsonl` 文件，不影响规范 Markdown 数据。
