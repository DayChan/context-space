## Context

当前 `LarkCliCommandRunner` 只允许四类业务读取命令，并在命令失败后解析权限 issue；首次同步因此会先创建运行并逐来源失败。`lark-cli` 已提供不读取业务数据的 `auth check --scope ... --json`，适合在同步边界前检查用户 token 的最小 scope。定时同步默认关闭，但手动 API 与调度器最终都调用同一 `LarkSyncService.sync()`，后端必须形成不可绕过的约束。

## Goals / Non-Goals

**Goals:**

- 在业务请求前得到结构化、可测试、可展示的权限状态。
- 首次完整同步前缺权时保持数据库、游标和分析队列无副作用。
- 权限撤销后仍保留既有来源级故障隔离，不让单一来源阻断其他增量数据。
- 只请求当前四类读取能力需要的最小 scope，并生成仅包含缺失项的授权命令。

**Non-Goals:**

- 自动执行 `lark-cli auth login`、打开飞书控制台或修改应用权限。
- 用 scope 预检替代日历访问角色等资源级权限判断。
- 改变来源分页、游标、标准化或分析队列语义。

## Decisions

### 使用独立权限检查器而非放宽业务命令 Runner

新增可注入的权限检查契约，生产实现只执行固定的 `lark-cli auth check --scope <固定列表> --json`。现有业务 Runner 会自动附加 `--as user` 和 `--format json`，这些参数不适用于 `auth check`；强行复用会扩大命令白名单并混合两种输出契约。独立检查器保持参数固定、`shell=false`、超时和输出上限，并便于测试 CLI 缺失、认证失败、缺失 scope 与非法输出。

### 后端每次预检，首次失败形成硬阻断

`LarkSyncService.sync()` 每次运行都先检查权限，以发现 token 或 scope 变化。`SyncRepository` 增加是否存在 `succeeded` 运行的只读查询：不存在时，任何未就绪状态都在 `startRun` 前抛出结构化预检错误；存在时记录并公开告警，但继续既有来源调用，由真实 API 响应决定每个来源是否失败。

相比仅在前端检查，此方案覆盖直接 API 和定时同步；相比每次缺权都全局阻断，它保留系统已经建立的增量与部分成功能力。

### 预检状态独立于同步运行

预检结果属于当前 CLI/认证环境，不伪装成五个来源之一，也不写入失败同步运行。服务缓存最近一次结果供状态 API 和 Settings 展示，字段包含状态、四项固定 required scopes、granted/missing scopes、检查时间、是否已完成首次同步、提示和授权命令。应用重启后可重新检查，不需要数据库 migration。

### UI 先检查再决定是否发送同步请求

Settings 加载独立预检 API，并提供“重新检查权限”。首次同步未就绪时同步按钮不可执行并展示缺失 scope 与复制友好的授权命令；完成过完整同步后只告警、不禁用按钮。后端仍重复执行预检，避免 UI 状态过期形成绕过。

## Risks / Trade-offs

- **[scope 齐全但资源仍不可读]** → 保留现有来源级结构化错误；日历 reader/writer/owner 等资源权限不由 scope 预检承诺。
- **[CLI 版本不支持 `auth check --json`]** → 返回检查失败与升级/排查提示，不把未知输出误判为已授权。
- **[检查结果在点击后变化]** → 前端预检只用于交互，后端在真正同步前再次检查。
- **[后续缺权仍调用部分业务接口]** → 这是为保留成功来源增量的有意兼容；Settings 会在调用前展示告警，实际失败来源不推进游标。
- **[授权命令泄漏不必要 scope]** → 命令从固定 required scopes 与实际 missing 集合求交生成，不接受服务端外部输入。

## Migration Plan

1. 部署权限检查器、状态类型和 API，不改变已有同步数据。
2. 接入同步服务首次成功判定与后端阻断。
3. 开放 Settings 状态和重新检查交互。
4. 回滚时移除预检调用即可，SQLite 无新增结构需要回退。

## Open Questions

无。最小 scope、首次阻断、后续来源级兼容和不自动授权均已确定。
