## ADDED Requirements

### Requirement: 机器数据使用 SQLite 规范存储
系统 SHALL 使用工作区内的单个 SQLite 数据库规范存储飞书来源、上游任务与人物身份、同步状态、运行配置、分析任务与运行、候选、证据和接受操作，并 MUST NOT 为新机器数据创建规范 Markdown 文档。

#### Scenario: 持久化一条飞书来源
- **WHEN** 适配器返回一条具有稳定提供方 ID 的有效来源
- **THEN** 系统在 SQLite 中创建或更新唯一来源记录，而不创建来源 Markdown

### Requirement: 数据库迁移与事务
系统 MUST 按版本顺序执行幂等数据库 migration、启用外键并使用事务维护跨表不变量；数据库初始化失败时 MUST 在接受写请求前停止启动。

#### Scenario: 首次打开空工作区
- **WHEN** 应用在没有数据库的有效工作区启动
- **THEN** 系统创建权限受限的数据库、执行全部 migration，并提交完整 Schema

#### Scenario: 事务中途失败
- **WHEN** 来源写入成功但同一事务内的分析任务入队失败
- **THEN** 整个事务回滚，来源和任务均不可见

### Requirement: 原始正文保留期
系统 SHALL 默认保存原始来源正文 90 天并允许用户配置保留天数；清理后 MUST 保留来源 ID、发生时间、参与者、正文摘要哈希和必要审计元数据。

#### Scenario: 清理已完成来源
- **WHEN** 一条来源超过保留期、已完成分析且不被待审核候选阻塞
- **THEN** 系统删除其原始正文、记录清理时间并保留元数据

#### Scenario: 延迟清理待审核证据
- **WHEN** 超过保留期的来源仍被待审核候选引用
- **THEN** 系统保留其必要正文，直到候选审核完成

### Requirement: 非破坏性旧数据迁移
系统 SHALL 幂等导入旧机器 Markdown，生成包含成功、跳过、冲突和失败数量的迁移报告，并 MUST NOT 在未经用户确认时删除原文件。

#### Scenario: 重复执行迁移
- **WHEN** 同一旧工作区迁移执行两次
- **THEN** 第二次运行根据稳定 ID 跳过已导入数据，且不会创建重复记录
