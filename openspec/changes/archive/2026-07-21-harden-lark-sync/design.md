## 背景

飞书适配器目前直接使用 `Date.toISOString()` 构造消息查询时间，因此会生成带毫秒的值。实测 `im +messages-search` 在 `lark-cli 1.0.72` 下仍会把该值原样传给飞书，并收到 `99992402 field validation failed`；去掉毫秒后同一查询成功。任务命令未传 `--complete=false`，标准化逻辑也会接收已完成任务。

`LarkCliCommandRunner` 使用 `execFile`，但子进程失败时没有解析 `stderr` 中的 JSON 信封。权限类型、缺失 scope、飞书控制台链接、排查链接和 `_notice.update` 因此都被折叠进 Node.js 的通用命令错误。同步 API 又以 HTTP 200 返回部分失败状态，Settings 当前仍提示“只读同步完成”。

## 目标与非目标

**目标：**

- 只请求并接收未完成的飞书任务。
- 为消息和日历查询生成飞书可接受的秒级 ISO 时间。
- 将 `lark-cli` 错误转换为稳定、可持久化、可由 UI 展示的诊断结构。
- 对权限、认证和升级问题提供清晰的人工处理入口。
- 保持来源级故障隔离和只读安全边界。

**非目标：**

- 不自动执行 `auth login`、打开权限、升级 CLI 或重试不可逆操作。
- 不删除工作区中历史上已经保存的已完成任务。
- 不改变飞书来源的同步频率或增加后台调度器。
- 不改变 LLM 分析流程。

## 决策

### 使用秒级 UTC ISO 时间

查询时间由统一函数生成，先截断毫秒，再输出 `YYYY-MM-DDTHH:mm:ssZ`。该格式已通过当前飞书消息搜索实测，并且不引入本地时区歧义。

备选方案是改用本地时区偏移格式，但这需要额外的时区转换依赖或手写格式化，且不能比 UTC 秒级格式提供更多信息。

### 请求端与标准化端双重过滤任务

任务命令增加 `--complete=false`，减少传输和处理数量；`normalizeTasks` 同时丢弃上游仍意外返回的已完成记录，避免 CLI 版本差异破坏业务约束。

该变更不会把“本次未返回”解释为“上游已经完成”，因为缺失也可能来自分页、权限或暂时故障。历史 Todo 的上游完成状态协调留给后续单独设计。

### 解析并持久化结构化问题

Runner 将解析成功或失败信封中的 `error` 与 `_notice.update`。子进程退出失败时优先读取 `stderr` JSON；无法解析时才回退为通用命令错误。错误对象包含：

- 问题类别与是否需要人工处理；
- 飞书错误代码、类型和子类型；
- `missing_scopes`、`hint` 与 `console_url`；
- `log_id` 与 `troubleshooter`；
- CLI 当前版本、最新版本和升级命令。

适配器把该对象映射到来源同步结果，旧的 `error` 字符串继续保留，确保 API 向后兼容。

### UI 只提醒，不替用户执行处理命令

Settings 对权限或认证问题展示“需要处理”提醒，展示缺失 scope、官方提示和安全链接；参数错误展示错误代码与排查入口；CLI 升级通知展示官方升级命令。页面不得自动触发授权或升级。

同步调用返回部分失败时，前端根据返回状态提示“同步完成，但存在失败来源”，而不是显示无条件成功。

## 风险与权衡

- **风险：不同 `lark-cli` 版本的错误信封字段可能变化** → 采用宽松读取、稳定内部字段和通用错误回退，并用多种信封测试覆盖。
- **风险：仅拉未完成任务后无法观察任务刚刚完成的状态** → 本次严格遵循“排除已完成”的要求，不根据缺失记录自动改状态，避免误判。
- **风险：错误链接可能缺失或格式异常** → UI 仅把有效的 `http` 或 `https` URL 渲染为链接，其余内容按纯文本展示。
- **权衡：双重任务过滤存在重复逻辑** → 以较小复杂度换取跨 CLI 版本的一致行为。

## 迁移计划

1. 部署代码后，后续飞书同步自动使用秒级时间和未完成任务过滤。
2. 现有同步状态文件无需迁移；新增问题字段为可选字段。
3. 工作区中已经存在的已完成任务文件保持不变。
4. 如需回滚，可恢复旧版命令构造；新增状态字段会被旧版本忽略。
