## 背景

消息同步当前把一个七天窗口交给单次 `lark-cli im +messages-search --page-all`。该 shortcut 会在内部依次执行搜索、详情查询、会话富化和表情富化，直到达到最多四十页；应用只能在整个子进程退出后取得结果。实测首个 P2P 窗口在关闭表情富化后读取 2,000 条消息仍需 112 秒且 `has_more=true`，正式命令因此被 Runner 的 120 秒整体超时终止。

现有来源 Markdown 使用稳定消息 ID，重复写入已经幂等；来源检查点只在整个来源成功后推进。这两个约束允许应用逐页持久化并在失败后安全回放窗口，无需保存可能过期的飞书分页游标。

## 目标与非目标

**目标：**

- 由应用逐页驱动消息查询，完整消费 `has_more` 和 `page_token`。
- 将 120 秒超时约束在单页 CLI 调用，而不是整个七天窗口。
- 每页成功后立即持久化，后续页面失败时保留已取得的来源。
- 对缺失游标、重复游标和安全页数上限给出明确失败，禁止伪成功和静默截断。
- 保持来源级检查点、稳定 ID、两阶段批量分析和只读安全边界。

**非目标：**

- 不提高全局超时来容纳无边界的 `--page-all`。
- 不持久化飞书 `page_token`；其有效期没有稳定契约，不能作为长期恢复依据。
- 不改变初始回填天数、默认七天时间窗口或 LLM 批次容量。
- 不为消息表情建立领域模型。

## 决策

### 应用层每次只读取一页

`LarkAdapter.fetchSource` 对消息来源构造 `--page-size 50 --page-limit 1 --no-reactions`，并在后续调用附加上一页返回的 `--page-token`。适配器返回规范记录以及 `{hasMore, nextPageToken}`，同步服务负责循环。

选择应用层分页而不是继续使用 `--page-all`，因为应用必须观察每页边界、及时持久化、记录进度并判断是否完整。单纯提高超时仍受四十页上限约束，无法解决静默截断。

### 成功页立即落盘，失败窗口幂等回放

同步服务收到一页后立即写入来源和人物基础档案，再请求下一页。若后续页面失败，已写入记录保留，但当前来源不推进 `last_success_at`。下一次同步从检查点计算出的时间范围重新读取该窗口，已有消息按稳定来源 ID 更新而不重复创建。

不持久化 `page_token`。游标可能过期，并且跨进程恢复后直接跳过已落盘页面会使“全量采集后统一分析”看不到之前页面。回放窗口虽然增加少量读取成本，但恢复语义稳定，且分析仍能取得完整窗口记录。

### 显式验证分页完整性

同步服务维护当前窗口已见游标集合，并设置每窗口最多 200 页的安全上限。以下情况把窗口和来源标记为失败：

- `has_more=true` 但没有非空 `page_token`；
- 下一游标与当前游标相同，或此前已经出现；
- 达到 200 页后仍有下一页；
- 任一单页 CLI 调用失败或超时。

默认上限允许每个七天窗口最多读取 10,000 条消息。上限是防止异常游标造成无限循环的保险，不是完整性截断点；达到上限必须失败且不得推进检查点。

### 保持来源级检查点兼容

`.context/sync/lark.md` 继续只保存各来源的 `last_success_at`。只有一个来源的所有时间窗口和所有页面都完整结束后才推进来源检查点。已经存在的工作区无需迁移。

### 增加页级结构化日志

保留现有窗口日志，并增加页开始和页完成事件，记录来源、窗口序号、页序号、是否有下一页、接收数量、累计持久化数量和耗时。日志不得记录分页游标、消息正文或 CLI 响应。

## 风险与权衡

- **风险：窗口回放会重复读取已成功页面** → 稳定来源 ID 保证写入幂等；相比持久化不稳定游标，额外读取换取确定的恢复语义。
- **风险：单个窗口超过 10,000 条消息** → 明确报告分页未完成并保留已写入数据；后续可基于实际规模引入自适应时间切窗，而不是静默提高上限。
- **风险：逐页启动 CLI 增加固定进程开销** → 每页最多 50 条，避免一次聚合占用大内存；正确性和可观察性优先于少量启动成本。
- **风险：分页过程中消息集合变化导致页间漂移** → 现有重叠窗口和稳定 ID 可在后续同步中收敛；不把页序号当作永久检查点。

## 迁移计划

1. 部署后，新的消息窗口自动使用单页循环和 `--no-reactions`。
2. 现有 `.context/sync/lark.md`、来源文档和同步状态无需迁移。
3. 首次同步中失败的 P2P 来源没有成功检查点，下一次运行会重新回填并复用已有稳定来源文件。
4. 如需回滚，可恢复旧命令构造；新实现未写入旧版本无法识别的持久化字段。
