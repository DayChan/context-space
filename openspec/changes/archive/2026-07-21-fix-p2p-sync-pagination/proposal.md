## Why

首次同步按七天窗口一次性执行 P2P 消息全量分页；当窗口内超过 2,000 条消息时，`lark-cli --page-all` 会在仍有下一页时停止，而默认消息富化又会使整条命令超过固定的 120 秒超时。当前实现因此既可能把整个 P2P 来源标记失败，也可能在提高超时后静默遗漏消息，无法满足已有的有边界回填和增量幂等要求。

## What Changes

- 将消息同步从 CLI 内部全量聚合改为应用层受控的逐页读取，明确消费并校验 `page_token` 与 `has_more`。
- 每页成功后立即规范化并持久化来源；分页失败时保留已完成页面并把窗口标记为未完成，后续运行通过稳定来源 ID 安全回放该窗口。
- 消息同步显式关闭业务未使用的表情富化，避免额外请求占用同步预算。
- 当分页达到安全上限或游标异常时明确将窗口标记为未完成，不推进来源完成检查点，不允许静默截断。
- 为多页 P2P、分页中断恢复、游标异常、页数上限和重复同步补充自动化测试与结构化日志。

## Capabilities

### New Capabilities

无。

### Modified Capabilities

- `lark-context-sync`：将消息分页、窗口完成判定和检查点推进约束扩展为逐页持久化、可恢复且禁止静默截断。

## Impact

- 主要修改 `src/adapters/lark/adapter.ts`、`src/adapters/lark/sync.ts`、`src/adapters/lark/runner.ts` 及相关类型与测试。
- 飞书集成仍使用只读用户身份和现有 `lark-cli im +messages-search`，不新增写操作或外部依赖。
- 现有来源 Markdown、稳定来源 ID 和来源级检查点保持兼容，不持久化可能过期的飞书分页游标。
- 本地同步 API 的现有成功与失败字段保持兼容，未完成分页会作为明确失败而不是伪成功返回。
