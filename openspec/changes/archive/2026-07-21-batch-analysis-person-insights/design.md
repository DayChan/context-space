## 背景

当前同步器在每条记录落盘后立即调用 `AnalysisCoordinator.analyze`。这种交错流程使每条消息都承担一次 Codex 启动、固定 Prompt 和结构化输出成本，也让模型看不到同一轮同步中的前后消息。人物档案虽然已有 `observations` 字段，但没有 LLM 生成和协调机制。分析配置已有 `model` 字段，两种 Codex Provider 也会传递该字段，Settings 尚未暴露它。

失败记录中的 `tool_activity` 还有一处独立问题：当前实现只允许 `agent_message` 和 `reasoning`，而 SDK 的合法无副作用项目还包括 `todo_list` 和非致命 `error`。因此，当前白名单会把“不是响应文本”的项目等同于“执行了工具”，并且 Provider 抛错后事件类型没有传回运行记录。

## 目标与非目标

**目标：**

- 一轮同步先完成所有可用来源的拉取和规范来源落盘，再执行 LLM 分析。
- 在安全容量内用尽可能少的请求携带尽可能多的按时间排序上下文。
- 让 Todo、知识和人物洞察都具有可机器校验的逐条来源证据。
- 将职责和职场协作方式写回人物档案，同时保留人工字段和现有观察。
- 为 SDK 与 Exec 提供一致的显式模型选择。
- 只拒绝真实工具活动，并把实际事件类型写入失败诊断。

**非目标：**

- 不把无限量历史或整个工作区无边界地塞进单个模型上下文。
- 不生成心理诊断、人格类型、受保护属性、绩效判断或招聘适配结论。
- 不自动改写人工角色、人工观察、Leader 设置或用户编辑的派生文档字段。
- 不枚举或承诺账户可用的模型列表；模型可用性由当前 Codex 认证与服务端校验。

## 技术决策

### 同步采用“采集落盘”和“分析派生”两个阶段

同步器先完成各来源窗口的抓取、去重、来源与人物基础档案写入，然后重建一次索引，最后把本轮成功取得的记录交给批量分析器。单个飞书来源失败仍不阻塞其他来源和已取得记录；LLM 失败也不回滚来源文档或检查点。

选择这一方案而不是在每个来源结束后分析，是因为跨来源上下文可帮助识别同一人的职责和承诺，并能保证“拉完再分析”的可观察语义。原生任务映射仍为确定性逻辑，但也在采集阶段之后统一执行。

### 使用确定性的有界贪心批次

分析器按 `occurred_at` 和稳定来源 ID 排序、按来源 ID 去重，再依次装入批次。批次同时受 `max_batch_records` 与 `max_batch_source_chars` 限制，每条来源仍受 `max_source_chars` 限制。默认值分别为 50 条、60000 字符和 20000 字符；不足上限时整轮消息只产生一次 LLM 请求，超限时才拆分。

选择字符和记录双上限而不是声称精确计算模型 Token，是因为 Provider 可选择不同模型与分词器，字符预算更稳定、可测试，也不会依赖额外 Tokenizer。批次边界及所有来源哈希进入运行标识，完全相同的成功批次可直接跳过。

### 输出契约升级为逐来源证据

`work-context/analysis@2` 中，Todo 和知识使用 `source_refs`，证据改为 `{source_ref, quote}`。新增 `person_insights`，包含稳定人物 ID、受限类别、文本、置信度、证据和简短依据。校验器确保：

- 每个引用都属于当前批次；
- 每段引文都能在发给模型的截断后正文中逐字定位；
- Todo 相关人员和人物洞察目标都来自当前批次参与者；
- 非职责型协作风格至少由两条不同来源支撑；
- 结果在整体校验通过前不产生任何派生写入。

### 人物洞察采用分类观察而不是覆盖角色

人物观察类别限定为 `responsibility`、`communication_style`、`collaboration_style` 和 `work_preference`。生成观察保存 `insight_key`、引用来源、逐字证据、置信度、观察时间和 `origin=inferred`。同一证据与类别使用稳定键更新，人工观察始终原样保留。

“personality”在产品中表达为有范围的职场协作方式，不写成不可变人格标签。职责也作为可追溯观察保存，不自动覆盖通讯录或人工维护的 `role`。

### Codex 事件校验采用真实工具类型拒绝集

明确允许 `agent_message`、`reasoning`、`todo_list` 和 `error` 这些不产生外部副作用的项目，拒绝 `command_execution`、`file_change`、`mcp_tool_call`、`web_search` 及未知的新项目类型。最终响应仍必须通过大小、JSON Schema 和领域校验。拒绝时 `AnalysisProviderError` 携带完整事件类型，运行记录显示实际工具类型，而不是空数组和笼统信息。

这一决策不降低沙箱边界：Provider 仍关闭 Shell、MCP、应用、网络和 Web 搜索，并在只读临时目录运行。事件检查是第二道结果拒绝机制，而不是唯一防线。

### 模型选择复用统一分析配置

Settings 提供可留空的模型输入。留空保存为 `null`，由 Codex 使用当前推荐默认模型；非空值原样进入分析运行配置快照。`codex-sdk` 传给 `startThread({ model })`，`codex-exec` 传给 `--model`。服务不静默回退模型，账户不可用或模型标识无效时按 Provider 错误记录。

## 风险与权衡

- [批次越大，单次失败影响的来源越多] → 对批次设置双上限，逐批隔离失败，并在重分析接口中复用相同批处理逻辑。
- [新增消息会改变批次边界并降低运行跳过率] → 使用稳定排序与来源哈希；后续可在不改变契约的情况下增加逐来源分析指纹。
- [模型可能从有限证据过度概括一个人] → 限制类别、要求引文与置信度、对风格要求多来源证据，并在 UI 中明确标记为推断。
- [不同模型的上下文能力不同] → 使用保守可配置字符上限，不根据模型名称硬编码上下文窗口。
- [允许 `error` 项可能掩盖模型内部非致命问题] → 仍保存事件类型和用量；只有最终结构化响应完整通过校验时才接受业务结果。

## 迁移方案

1. 初始化工作区与配置读取使用新默认字段补齐旧 `config/analysis.md`，保存配置时写出完整新字段。
2. Prompt 升级为 `context-analysis@2`，输出升级为 `work-context/analysis@2`；旧运行和旧派生文档继续可读。
3. 人物观察新增字段均为兼容性可选字段，现有人工与旧推断观察无需迁移。
4. 如需回滚，可恢复旧 Prompt 与协调器；规范来源和已有人物观察仍是普通 Markdown，不影响来源数据恢复。

## 开放问题

无。模型列表枚举和按模型自动预算留待后续独立变更。
