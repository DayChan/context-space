## Why

当前提及和 P2P 消息在拉取过程中逐条调用 LLM，重复承担启动与固定 Prompt 成本，也无法利用同一批工作上下文判断跨消息职责、协作方式和承诺。与此同时，Codex SDK 的非工具事件会被过宽的事件白名单误判为 `tool_activity`，模型配置虽然已存在于后端却无法在界面中管理。

## What Changes

- 将同步流程改为先完成所有来源拉取、规范化和持久化，再对可分析消息进行有上限的上下文批处理。
- 在单次请求中尽量装入更多按时间排序的消息，同时通过可配置的单条和整批容量上限避免超过模型上下文。
- 扩展结构化分析结果，使 Todo、知识和人物洞察可引用同一批次中的一条或多条来源。
- 将有证据、置信度和时间戳的职责、沟通方式、协作方式及工作偏好写回人物档案；禁止敏感属性、诊断式标签和对人工内容的覆盖。
- 为 `codex-sdk` 与 `codex-exec` 增加统一的模型选择界面和配置快照。
- 将 Codex 事件校验改为明确拒绝真实工具活动，允许无副作用的响应、推理、计划和错误事件，并在失败运行中保存实际事件类型和可操作诊断。

## Capabilities

### New Capabilities

无。

### Modified Capabilities

- `llm-content-analysis`: 从逐条分析改为有边界的批量上下文分析，扩展输出契约、模型选择和工具事件诊断。
- `lark-context-sync`: 将 LLM 分析阶段移动到所有来源拉取和持久化之后。
- `people-profiles`: 增加由 LLM 生成并可追溯的人物职责与职场协作洞察写回机制。
- `context-workbench-ui`: 增加模型配置入口，并展示人物洞察及其来源依据。

## Impact

主要影响 `src/analysis/` 的 Prompt、Schema、校验、运行记录和写入器，`src/adapters/lark/sync.ts` 的同步阶段划分，人物 Markdown 元数据，Settings 与人物详情界面，以及相关 API、测试和使用文档。分析 Schema 与 Prompt 版本将升级，旧来源和旧运行记录继续可读；新的分析运行使用新版本契约。
